## 📝 完整讲稿（按时间分段）

### 【第1段：项目背景】（0-30秒）

```
"我做的项目是一个企业知识助手Agent系统，基于Spring Boot和LangChain4j。

核心问题是：企业员工经常有关于公司政策的问题——比如'北京的差旅报销标准是什么'，
我们需要一个智能助手，用LLM来理解这些问题，然后通过查询内部文档、调用各种工具来
回答。

这个项目的核心是如何设计一个**可观测、可控、可测试的Agent执行架构**。

让我给你展示一下我是怎么设计这个系统的。"
```

**关键词**：
- ✅ 明确说出项目名称和技术栈
- ✅ 解释解决的实际问题
- ✅ 预告核心设计理念

---

### 【第2段：核心架构介绍】（30秒-1分30秒）

```
"这个系统采用分层架构设计，分为5层。

【拿起笔，开始画第一张图】

最上面是API层 - 我提供了两种API：
  - POST /api/chat 同步接口，简单直接
  - GET /api/chat/stream 流式接口，可以实时推送事件

中间是我最核心的编排层。这一层实现了完整的Plan-Act-Observe-Reflect循环。
我用一个循环，最多执行3步，每一步都有超时控制和重试机制。

下面是执行层，用来处理工具的调用，确保超时、异常都被正确处理。

再下面是工具层，这里定义具体的工具 - 比如searchPolicy查询政策文档。

最下面是数据层 - PostgreSQL配合PgVector来存储文档和向量。

这样分层的好处是：
1. 关注点分离，每层职责清晰
2. 易于扩展 - 要加新工具只需在工具层添加
3. 易于测试 - 可以独立测试每一层
"
```

**关键词**：
- ✅ 清晰描述5层
- ✅ 突出Plan-Act-Observe-Reflect
- ✅ 解释分层的优势

---

### 【第3段：详细讲解 - 图1（分层架构）】（1分30秒-2分30秒）

```
"现在看这张分层架构图。

【指向图1】

我想特别强调的是中间这个编排层。它是整个系统的大脑。

当一个请求进来时，编排层会：
1. Plan阶段 - 分析问题
2. Act阶段 - 执行工具调用，有15秒的超时控制
3. Observe阶段 - 记录执行结果到AgentStep
4. Reflect阶段 - 如果失败了，调整策略后重试

关键设计点是这个循环最多执行3步，防止无限循环和成本失控。

我用MDC来实现分布式追踪 - 每个请求都有一个唯一的traceId，
这样所有的日志都会自动带上这个traceId，非常便于调试。

执行层负责处理超时和重试。我用Java Future.get()来实现超时控制，
实际执行一个工具最多会尝试两次 - 第一次如果超时，就重试一次。

这样的设计保证了系统的可控性。"
```

**关键词**：
- ✅ 逐个层次说清楚
- ✅ 突出Plan-Act-Observe-Reflect
- ✅ 强调超时控制和重试
- ✅ 提到MDC追踪

---

### 【第4段：详细讲解 - 图2（时序图）】（2分30秒-4分钟）

```
"现在看第二张图，这是一个完整请求的处理流程。

【指向图2】

用户问'北京的差旅报销标准是什么'为例。

首先ChatController检查Token配额，防止滥用。

然后进入编排层，在这一刻，我设置MDC，创建AgentRun对象。

Plan阶段很简单，分析一下问题。

Act阶段是关键 - 调用ToolExecutionWrapper，它会：
- 通过反射调用@Tool注解的searchPolicy方法
- 用Future.get(15秒)实现超时控制
- LLM决策需要这个工具，就调用它，查询知识库

一旦得到工具结果，进入Observe阶段，记录这个步骤的所有信息：
- stepId、耗时、输入输出摘要、状态

然后LLM根据搜索结果生成答案。

关键问题来了 - 如果这一步失败了呢？那就会进入Reflect阶段。
我会告诉LLM'前一步失败了，请直接根据你的知识回答'。
然后再试一次。最多三步。

最后清理阶段 - 必须要调用MDC.clear()，避免内存泄漏。

最后返回给前端一个SSE事件流。"
```

**关键词**：
- ✅ 用具体例子走流程
- ✅ 强调关键节点（Token检查、MDC设置、超时控制）
- ✅ 解释失败时的处理（Reflect阶段）
- ✅ 提到资源清理

---

### 【第5段：图3讲解（重试和反思）】（4分钟-4分30秒）

```
"如果想深入看失败时发生了什么，看这张图。

【指向图3】

第1步可能超时了，那么重试机制被触发。
还是超时？好吧，那么这一步就算失败了。

进入Reflect阶段，我们调整策略。
不让LLM继续调用工具，而是让它基于已有知识直接回答。

第2步用新的提示词重试，这次成功了。

这就是系统的自我恢复能力。"
```

**关键词**：
- ✅ 简洁快速
- ✅ 展现容错精神

---

### 【第6段：技术亮点】（4分30秒-5分钟）

```
"我这个设计的核心亮点有三个：

一、**可观测性**
每个步骤都被完整记录。用户和开发者都能看到Agent在做什么。
通过MDC + AgentRun/AgentStep，形成了完整的执行轨迹。
支持includeTraceInResponse配置，开发环境详细，生产环境简化。

二、**可控性**
通过maxSteps、timeout、retryTimes等参数，精细控制系统行为。
防止无限循环、成本失控、资源浪费。

三、**可测试性**
分层设计让每一层都可以独立测试。
我写了8个单元测试，覆盖核心逻辑，测试通过率100%。

这三个'可'就是我设计的核心理念。"
```

**关键词**：
- ✅ 总结核心价值（可观测、可控、可测试）
- ✅ 用具体数据支持（8个单元测试）
- ✅ 有力收尾

## 🎯 讲解技巧

### ✅ 要做的事

1. **边讲边画**
   - 不要一开始就把所有图都画出来
   - 按照讲稿的进度逐步绘制
   - 保持听众的注意力跟着你

2. **用具体例子**
   - 不要说"系统会执行Plan"
   - 要说"用户问'北京的差旅报销标准'，系统先分析这个问题"

3. **强调关键数字**
   - maxSteps = 3
   - timeout = 15秒
   - retryTimes = 1
   - 8个单元测试

4. **用视觉强调**
   - 重点用颜色标记（如果在白板上）
   - 用手指指向关键部分
   - 用身体语言强调重要性

### ❌ 要避免的事

1. ❌ 陷入过度细节
   - 不要讲某个方法的具体实现
   - 除非面试官特别问

2. ❌ 一次性讲完所有东西
   - 5分钟讲40张图是自杀
   - 抓住最核心的4张

3. ❌ 没有逻辑跳跃
   - 每一段都要有清晰的承接
   - "现在让我详细讲一下..." 
   - "这特别重要，所以我想多说两句..."

4. ❌ 忽视反馈
   - 看面试官的表情
   - 如果他/她有困惑，主动澄清
   - 如果他/她兴趣浓厚，可以适当展开
