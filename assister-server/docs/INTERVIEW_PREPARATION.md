### ✅ 参考答案

**候选人**：好的，让我详细介绍一下这个Agent系统的架构。

#### 整体架构

这是一个基于Spring Boot和LangChain4j的企业级Agent系统，采用**分层架构**设计，从外到内分为5层：

**1. API层（ChatController）**
- 提供RESTful API和SSE流式接口
- 处理HTTP请求、参数验证、权限控制
- 支持同步（/api/chat）和流式（/api/chat/stream）两种响应模式
- 集成Token配额检查，防止滥用

**2. 编排层（AgentOrchestratorService）**
- 这是核心层，实现了**Plan-Act-Observe-Reflect循环**
- 控制Agent执行流程，最多执行maxSteps步（默认3步）防止无限循环
- 提供超时控制（每步最多15秒）和重试机制（失败后重试1次）
- 使用SLF4J的MDC实现分布式追踪，每个请求有唯一的traceId
- 记录完整的执行轨迹到AgentRun对象

**3. 执行层（ToolExecutionWrapper）**
- 统一工具调用的执行方式
- 使用反射动态调用@Tool注解的方法，保持与LangChain4j框架的兼容性
- 提供超时控制（使用Future.get实现）和重试机制
- 返回结构化的ToolExecutionResult（成功/失败、耗时、错误信息）

**4. 工具层（AgentTools）**
- 定义具体的工具方法，如：
  - searchPolicy：查询公司内部政策文档（集成混合检索）
  - getWeather：查询天气信息
  - sendEmail：发送邮件（需要人工确认）
- 使用LangChain4j的@Tool注解声明工具
- 集成了混合检索（向量检索+全文检索）

**5. 数据模型层（AgentRun/AgentStep）**
- AgentRun：记录完整的执行轨迹
  - runId（UUID，用作traceId）
  - username（用户名，用于审计）
  - userQuestion（原始问题）
  - steps（执行步骤列表）
  - status（RUNNING/SUCCESS/FAILED）
  - totalLatencyMs（总耗时）
- AgentStep：记录单个步骤的详细信息
  - stepId（步骤唯一标识）
  - sequence（步骤序号）
  - toolName（工具名称）
  - inputSummary/outputSummary（输入输出摘要）
  - status（PLANNED/RUNNING/SUCCESS/FAILED/TIMEOUT）
  - latencyMs（步骤耗时）
  - errorCode/errorMessage（错误信息）

#### 数据流示例

当用户问"北京的差旅报销标准是什么？"时：

1. **API层**：ChatController接收请求，检查Token配额
2. **编排层**：AgentOrchestratorService创建AgentRun，设置MDC上下文
3. **Plan阶段**：分析用户问题（当前实现为直接返回，未来可扩展）
4. **Act阶段**：
   - 创建AgentStep，调用PolicyAgent
   - LLM决策需要调用searchPolicy工具
   - ToolExecutionWrapper执行工具，查询知识库
5. **Observe阶段**：记录工具执行结果到AgentStep
6. **判断**：如果LLM返回了最终答案，标记成功并返回
7. **返回**：ChatController返回ChatResponse给前端，包含：
   - 答案内容
   - 完整执行轨迹（runId、步骤列表、耗时）

#### 设计亮点

1. **可观测性**：
   - 使用MDC实现请求级追踪，所有日志自动带上traceId和username
   - 记录每个步骤的详细信息（输入、输出、耗时、错误）
   - API可返回完整执行轨迹，便于调试和分析

2. **可控性**：
   - 通过配置控制maxSteps、timeout、retryTimes，防止失控
   - 支持配置是否返回执行轨迹（开发环境详细，生产环境简化）

3. **可测试性**：
   - 核心组件都有单元测试，使用Mockito模拟依赖
   - 8个单元测试覆盖核心逻辑，测试通过率100%

4. **扩展性**：
   - 使用反射动态调用工具，新增工具无需修改代码
   - Plan-Act-Observe-Reflect循环易于扩展（如增加问题分解、工具选择等）

### 🎯 面试官追问

#### 追问1：编排层的工作流程

**面试官**：你提到了"agent编排"这一层，能具体说说这一层是如何工作的吗？比如，当一个用户请求进来，编排层会做哪些事情？

**候选人实际回答**：

按照以下流程执行：
1. 记录用户请求
2. 尝试分析问题
3. 开始执行agent循环
4. 以可重试的方式执行工具调用，会标记超时和失败的异常
5. 记录当前步骤的执行结果
6. 对于失败的步骤会给出反思的提示词
7. 检查是否到达了最大执行步数限制
8. 最后清理MDC避免内存泄漏和上下文污染

**面试官反馈**：很好！你把整个流程说得很清楚，特别是提到了MDC清理这个细节。

**参考答案（详细版）**：

好的，让我详细说明AgentOrchestratorService的工作流程。

当一个请求进来时，编排层会执行以下步骤：

**1. 初始化阶段**
```java
AgentRun run = new AgentRun(username, userMessage);
String traceId = run.getRunId();
MDC.put("traceId", traceId);  // 设置分布式追踪上下文
MDC.put("username", username);
```
- 创建AgentRun对象，生成唯一的runId
- 设置MDC上下文，后续所有日志都会自动带上traceId和username

**2. Plan阶段（规划）**
```java
String workingMessage = plan(userMessage);
```
- 当前实现为简单的直接返回
- 未来可扩展为：问题分解、工具选择、执行计划生成

**3. 执行循环（最多maxSteps步）**
```java
for (int i = 0; i < runtimeProperties.getMaxSteps(); i++) {
    // 创建步骤
    AgentStep step = run.newStep("policy_agent", summarize(workingMessage));
    step.start();

    // Act阶段：执行工具调用
    String answer = actWithRetry(step, workingMessage, systemPrompt);

    if (answer != null) {
        // Observe阶段：记录成功结果
        observe(step, answer);
        run.markSuccess(answer);
        return run;
    }

    // Reflect阶段：反思失败，调整策略
    workingMessage = reflect(userMessage, run);
}
```

**4. Act阶段（执行）**
- 调用`actWithRetry`方法，带重试机制
- 使用Virtual Thread执行器提交任务
- 使用Future.get(timeout)实现超时控制
- 如果超时或失败，会重试（最多retryTimes次）

**5. Observe阶段（观察）**
- 记录执行结果到AgentStep
- 标记步骤状态（SUCCESS/FAILED/TIMEOUT）
- 记录输出摘要和执行时间

**6. Reflect阶段（反思）**
- 如果当前步骤失败，分析原因
- 调整下一步的输入消息
- 当前实现：提醒LLM之前失败了，要求提供直接答案
- 未来可扩展：分析失败原因、调整工具选择、简化问题

**7. 保护机制**
- 如果达到maxSteps仍未成功，创建guard步骤
- 标记为MAX_STEPS_EXCEEDED错误
- 防止无限循环和成本失控

**8. 清理阶段**
```java
finally {
    MDC.clear();  // 清理MDC，避免内存泄漏
}
```

**关键设计点**：
- **超时控制**：每步最多15秒，防止LLM响应过慢
- **重试机制**：失败后重试1次，提高成功率
- **步数限制**：最多3步，防止无限循环
- **可观测性**：每个阶段都有详细日志
- **状态管理**：AgentRun和AgentStep记录完整状态

#### 深入追问：重试、反思、maxSteps

**追问1.1：重试策略**

**面试官**：你提到"以可重试的方式执行工具调用"，能具体说说重试策略吗？什么情况下会重试？重试几次？为什么不用指数退避？

**候选人实际回答**：

重试的次数是根据agent配置的重试属性决定的，包括重试执行中的超时时间设置。超时会抛出TimeoutException，标记当前步骤失败。当所有重试都超时或失败时，会标记整个步骤为对应的超时和失败状态。

**面试官点评**：✅ 很好！你准确地说明了重试次数由配置决定，并且区分了超时和失败两种情况。

**补充说明**：

具体实现细节：
```java
// 配置项
agent.runtime.retryTimes=1  // 默认重试1次（总共执行2次）
agent.runtime.stepTimeoutMs=15000  // 每步超时15秒

// 重试逻辑
for (int attempt = 0; attempt <= runtimeProperties.getRetryTimes(); attempt++) {
    try {
        return invokeWithTimeout(message, systemPrompt);
    } catch (TimeoutException e) {
        // 超时异常：标记为TOOL_TIMEOUT
        lastErrorCode = AgentErrorCode.TOOL_TIMEOUT;
    } catch (Exception e) {
        // 其他异常：标记为TOOL_EXECUTION_FAIL
        lastErrorCode = AgentErrorCode.TOOL_EXECUTION_FAIL;
    }
}
```

**为什么不用指数退避？**
1. **LLM调用特点**：不像网络请求，LLM超时通常是因为问题复杂度高，而不是临时拥堵
2. **用户体验**：指数退避会导致等待时间过长（15秒 -> 30秒 -> 60秒）
3. **成本考虑**：每次调用都有token成本，简单重试更经济
4. **未来扩展**：可以根据错误类型选择策略（网络错误用指数退避，超时直接失败）

**追问1.2：反思机制**

**面试官**：你提到"反思的提示词"，能举个例子吗？具体是怎么调整的？为什么这样调整能提高成功率？

**候选人实际回答**：

反思是当当前步骤失败或超时时，会把问题和一段要求输出完整信息的提示词给到大模型，告知执行的具体信息。

**面试官点评**：✅ 正确！你理解了反思的本质是"给LLM更多上下文信息"。

**具体实现**：

```java
private String reflect(String originalQuestion, AgentRun run) {
    return """
            User question: %s
            Previous execution failed. Please provide a direct, complete answer.
            """.formatted(originalQuestion);
}
```

**反思策略的作用**：
1. **提醒LLM之前失败了**：避免LLM重复相同的错误
2. **要求直接答案**：引导LLM不要再尝试调用工具，直接基于已有信息回答
3. **保留原始问题**：确保LLM不会偏离用户意图

**为什么能提高成功率？**
- 第1步：LLM可能尝试调用工具但失败
- 第2步：反思后，LLM知道工具不可用，会尝试基于通用知识回答
- 第3步：如果还失败，说明问题确实无法回答

**未来可扩展的反思策略**：
1. **分析失败原因**：解析错误信息，告诉LLM具体哪里出错了
2. **调整工具选择**：建议LLM尝试其他工具
3. **简化问题**：将复杂问题分解为子问题
4. **提供示例**：给出类似问题的成功案例

**追问1.3：maxSteps限制**

**面试官**：为什么要设置最大执行步数限制？如果没有这个限制会发生什么？为什么选择3步作为默认值？

**候选人实际回答**：

这是一种防范机制。对用户而言，可以避免无效的token消耗和不合理的使用体验；对于系统来说，则是出于功能性和安全性的考量，优化体验以及避免过长的上下文导致模型注意力涣散甚至撑爆上下文窗口，产生低效甚至无效输出。三步的设置会反过来要求用户对于自己提出的问题进行预先组织，可以更好地表达意图。合理的maxSteps应该根据问题的复杂程度、类型来决定。

**面试官点评**：🌟 优秀！你从**用户、系统、模型**三个维度分析了maxSteps的作用，特别是"反过来要求用户预先组织问题"这个观点很有洞察力！

**详细分析**：

**1. 如果没有maxSteps限制会发生什么？**
- **无限循环风险**：LLM可能陷入"调用工具 -> 失败 -> 再调用 -> 再失败"的循环
- **成本失控**：每步都消耗token，无限循环会导致巨额费用
- **用户体验差**：用户等待时间过长，甚至超时
- **上下文爆炸**：每步的输入输出都会累积到上下文中，最终超过模型窗口

**2. 为什么选择3步？**
- **经验值**：大多数问题在3步内可以解决
  - 第1步：尝试调用工具
  - 第2步：如果失败，反思后再试
  - 第3步：最后尝试，通常会直接给答案
- **成本平衡**：3步的token消耗在可接受范围内
- **用户体验**：3步大约需要30-45秒，用户可以接受

**3. 如何判断合理的maxSteps？**

根据问题类型调整：
- **简单查询**（"今天天气"）：1-2步足够
- **中等复杂**（"差旅报销标准"）：2-3步
- **复杂分析**（"分析5年数据"）：需要问题分解，不适合单个Agent

**4. 实际案例**

```
用户问题："北京的差旅报销标准是什么？"

Step 1:
  - LLM决策：调用searchPolicy工具
  - 工具返回：政策文档内容
  - LLM生成答案：成功 ✅

用户问题："帮我预订明天去上海的机票"

Step 1:
  - LLM决策：调用bookFlight工具
  - 工具执行：失败（没有这个工具）

Step 2:
  - 反思：之前失败了，请直接回答
  - LLM回答：抱歉，我无法预订机票，请访问... ✅

用户问题："分析公司5年的差旅数据"

Step 1-3:
  - 都失败（问题太复杂）
  - 达到maxSteps限制
  - 返回：问题过于复杂，建议分解为子问题 ❌
```

**5. 设计权衡**

| 维度 | maxSteps=1 | maxSteps=3 | maxSteps=10 |
|------|-----------|-----------|------------|
| 成功率 | 低 | 中 | 高 |
| 成本 | 低 | 中 | 高 |
| 响应时间 | 快 | 中 | 慢 |
| 用户体验 | 差 | 好 | 差（等待太久） |

**结论**：3步是一个经过权衡的合理值。

**实际场景追问**

**面试官**：如果用户问"帮我分析一下公司过去5年的所有差旅数据，找出异常模式"，这种复杂问题可能需要很多步骤。你会怎么处理？是增加maxSteps，还是有其他方案？

**建议回答思路**：
1. **不增加maxSteps**：这会导致成本和时间失控
2. **问题分解**：在Plan阶段将复杂问题分解为子问题
3. **多Agent协作**：使用专门的数据分析Agent
4. **异步处理**：将长时间任务放到后台，通过任务队列处理
5. **引导用户**：提示用户将问题分解为更具体的查询

**参考答案**：

我不会简单地增加maxSteps，而是采用以下策略：

1. **问题分解（推荐）**：
   - 在Plan阶段识别这是复杂问题
   - 分解为子问题：
     - "获取5年差旅数据"
     - "计算统计指标"
     - "识别异常模式"
   - 每个子问题用单独的Agent处理

2. **多Agent协作**：
   - 使用专门的DataAnalysisAgent
   - 该Agent有更高的maxSteps限制（如10步）
   - 有专门的数据分析工具

3. **异步任务**：
   - 将复杂分析放到后台任务队列
   - 返回taskId给用户
   - 用户通过轮询或WebSocket获取结果

4. **引导用户**：
   - 识别问题过于复杂
   - 返回：建议将问题分解为：
     - "过去5年差旅总支出是多少？"

**参考答案（详细版）**：

这个问题触及了Agent应用的一个重要UX设计理念。

**问题1：同步API的可观测性问题**

```
用户角度的时间线：
T=0s: 点击发送，看到加载动画
T=0.5s: 还在加载...（开始担心）
T=1.5s: 还在加载...（是不是卡了？）
T=2.5s: 还在加载...（服务故障了吧？）
T=3s: 收到结果（感觉像很久才回来，实际只有3秒）
```

**同步的缺点**：
- **用户不确定性**：看不到系统在做什么
- **焦虑感**：长时间无反馈导致用户疑虑
- **误诊**：用户可能手动刷新或重新发送（导致重复请求）
- **体验差**：感觉系统很慢，即使实际处理很快

**问题2：流式API的优势**

```
用户角度的时间线：
T=0s: 点击发送
T=0.1s: 收到 step_start 事件 "正在执行 policy_agent..."
       → 用户知道系统在处理 ✅
T=0.5s: 收到 step_complete "执行完成，耗时 450ms"
       → 用户知道进展 ✅
T=0.6s: step_start "分析结果..."
T=1.2s: 收到 token "北" 字
        token "京" 字
        token "市"...
       → 答案开始显示，用户看到进度 ✅
T=3s: 收到 complete 事件，全部结束
```

**流式的优势**：
1. **即时反馈**（降低焦虑）
   - 第一个事件在100毫秒内返回
   - 用户知道"系统在工作"

2. **过程透明**（提高信任）
   - 看到每个步骤的执行时间
   - 了解系统内部逻辑
   - 能识别是哪一步慢了

3. **实时指导**（优化答案质量）
   - 如果某步失败，可以推送ErrorEvent和建议
   - 用户可以在收到建议后调整提问
   - 例子：提示"问题过于宽泛，建议先查询基本信息"

4. **答案逐字显示**（视觉反馈）
   - 不是突然出现整个答案
   - 字符级流式输出增强了"持续进展"的感觉
   - 类似ChatGPT的效果

**问题3：为什么需要同时支持两种？**

| 维度 | /api/chat（同步） | /api/chat/stream（流式） |
|------|------------------|------------------------|
| **适用场景** | 移动应用、快速集成、后台任务 | Web应用、实时交互、命令行 |
| **网络开销** | 一次请求/响应 | 多次事件推送 |
| **用户体验** | 简单直接 | 互动丰富 |
| **复杂性** | 低 | 中 |
| **延迟要求** | 较宽松 | 严格 |

**具体情景**：
1. **移动App**：使用同步API
   - 网络连接可能不稳定
   - 屏幕小，事件流可能太多
   - 用户接受等待

2. **Web应用**：使用流式API
   - 浏览器原生支持SSE
   - 用户期望实时反馈
   - 可以展示进度条和过程日志

3. **CLI工具**：使用流式API
   - 实时输出步骤信息
   - 用户可以实时看到进展

4. **后台任务**：使用同步API
   - 直接将结果存储到数据库
   - 不需要前端交互

**问题5：实现细节**

```java
// 同步API：直接返回
@PostMapping("/chat")
public ResponseEntity<?> chat(@RequestBody ChatRequest request) {
    AgentRun run = agentOrchestratorService.run(username, request.message(), systemPrompt);
    return ResponseEntity.ok(ChatResponse.from(run, includeTrace));
}

// 流式API：异步推送事件
@GetMapping(value = "/chat/stream", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
public SseEmitter stream(@RequestParam String message) {
    SseEmitter emitter = new SseEmitter(60_000L);
    executor.submit(() -> {
        AgentRun run = agentOrchestratorService.run(username, message, systemPrompt);
        for (AgentStep step : run.getSteps()) {
            emitter.send(SseEmitter.event()
                .name("step_start")
                .data(new AgentStreamEvent.StepStartEvent(...)));
        }
    });
    return emitter;
}
```

**关键点**：
- 流式API使用Virtual Thread异步处理，不阻塞请求线程
- 即使一个用户的请求很慢，也不会影响其他用户
- SSE连接有超时机制（60秒），长时间无数据会自动断开

**设计权衡**：
- 流式模式增加了前端复杂性（需要处理事件、错误重连等）
- 但获得了更好的用户体验和互动能力
- 这是**用户体验 vs 复杂性**的经典权衡

### Q11: 为什么需要 maxSteps 限制？

**面试官提问**：为什么必须设置最大步数？为什么默认值通常取 3？

**参考答案（精简版）**：

`maxSteps` 是 Agent 的硬保护阀。没有这个限制，模型可能陷入"调用工具-失败-再调用"循环，导致响应时延失控、token 成本失控、上下文膨胀，最终答案质量下降甚至无法输出。  
默认值取 3 是工程折中：1 步成功率偏低，10 步成本和等待不可控；3 步通常覆盖"首次尝试 + 一次反思修正 + 一次兜底收敛"。  
线上会按任务类型动态调节，并用 `avg_steps`、`timeout_rate`、`success_rate` 持续校准。

### Q12: 工具执行失败怎么处理？

**面试官提问**：工具失败后你们怎么处理，是否直接结束？

**参考答案（精简版）**：

先做错误分型再决定动作：
- 超时/瞬时错误：有限重试（如 1 次）
- 参数错误/权限错误/工具不存在：不重试，快速失败

重试仍失败后进入降级策略：
- 对强依赖工具的任务（如下单、发邮件）：明确返回"无法完成动作"并给下一步建议
- 对非强依赖任务：基于已有上下文给有限答案，并明确不确定性

同时把 `errorCode/errorMessage/attempt` 记录到步骤轨迹，保证可观测与可追溯。

### Q13: SSE 流式响应为什么做？怎么实现？

**面试官提问**：请从体验、实现、异常处理三个角度说明。

**参考答案（精简版）**：

1. 用户体验：同步接口等待期间无反馈，用户容易误判卡死；SSE 能持续反馈步骤和结果，降低焦虑。  
2. 后端实现：基于 `SseEmitter` 推送事件，常见事件包括 `step_start`、`step_complete`、`token`、`complete`、`error`。  
3. 异常处理：出现工具失败/超时时，推送错误事件并携带错误码，不伪造成功；连接异常时做超时与资源清理。

**结合当前项目的真实状态（面试可加分）**：
- 已实现 SSE 与上述事件类型。  
- 当前是"后置流式"：先拿到完整 `AgentRun`，再推送步骤；`token` 是对最终答案逐字符发送，不是上游 LLM 原生实时 token。

### Q14: 配置管理怎么做？

**面试官提问**：哪些配置可调？如何区分环境？如何避免硬编码？

**参考答案（精简版）**：

1. 可调项：`maxSteps`、`stepTimeoutMs`、`retryTimes`、`includeTraceInResponse`、SSE 超时、模型与检索相关参数。  
2. 环境隔离：通过 `application.yml + application-{profile}.yml` 区分 dev/test/prod，启动时用 profile 切换。  
3. 防硬编码：使用 `@ConfigurationProperties` 统一收口运行参数，敏感信息（API Key/JWT/数据库密码）只走环境变量或密钥管理，不写入代码与仓库。  
4. 变更安全：关键配置设置默认值与边界校验（例如步数、超时上下限），避免误配置导致系统抖动。

## 当前项目主流程（按代码现状）

### 1) 主请求链路（/api/chat 与 /api/chat/stream）

1. `ChatController` 接收请求，先做 token 配额检查（`TokenMonitorService.isExceeded`）。  
2. 从 `PromptService` 获取系统提示词（DB 动态提示词优先，默认提示词兜底）。  
3. 进入 `AgentOrchestratorService.run`。  
4. 编排层先用 `LocalQueryRouter` 判断是否需要检索：  
   - 不需要：直接调用 `PolicyAgent.chat` 返回。  
   - 需要：先尝试 `OrchestratorAgent.plan` 生成 tasks。  
5. 若有 tasks：并行执行并聚合结果（weather 走内置逻辑，policy 走 `PolicyAgent.chat`）。  
6. 若 task 路径未得到结果：回退到循环执行（最多 `maxSteps`），每步 `actWithRetry + timeout + reflect`。  
7. 成功则返回 `AgentRun`，失败则标记 `MAX_STEPS_EXCEEDED`。  

### 2) SSE 现状

- 已使用 `SseEmitter` 推送 `step_start/step_complete/token/complete/error`。  
- 但当前是后置发送：`run` 完成后再遍历 steps 推送。  
- `token` 是对最终答案逐字符拆分，不是上游 LLM 原生实时 token。

### 3) HITL 现状

- `HumanInLoopAgent` 中实现了危险工具 `Y/N` 人工确认。  
- 但主 API 链路当前并未走 `HumanInLoopAgent`。  
- 审批审计、幂等防重尚未在主链路落地。

## Q21-Q40（项目相关统一调整版）

> 口径原则：先说“当前实现”，再说“风险边界”，最后说“下一步改造”。

### Q21: 如果 LLM 一直返回 null 怎么办？
- 当前实现：编排层有 `retry + timeout + maxSteps + reflect` 兜底。  
- 风险边界：连续空结果会触发 `MAX_STEPS_EXCEEDED`。  
- 下一步：增加空响应判定与熔断降级（直接返回结构化错误与建议）。

### Q22: 日志过多怎么办？
- 当前实现：有 traceId（MDC）与分层日志。  
- 风险边界：高并发下日志成本高、检索困难。  
- 下一步：按环境收敛日志级别，关键字段结构化输出，接入统一日志平台。

### Q23: 如何实现工具调用幂等性？
- 当前实现：主链路未落地幂等键；`sendEmail` 这类操作型工具未做防重。  
- 下一步：为操作型工具引入 `idempotencyKey + 去重存储 + 状态机`。

### Q24: 如果支持多模态怎么办？
- 当前实现：仅文本输入与文本检索。  
- 下一步：接入文件/图片解析管道，扩展向量化与 tool schema，按模态路由模型。

### Q25: 如何实现 A/B 测试？
- 当前实现：尚未内建实验分流。  
- 下一步：在编排入口按用户或会话做分桶，对比 success/latency/token 三类指标。

### Q26: 如何监控系统健康状态？
- 当前实现：已有 `/api/tools/health`、`/api/tools/dashboard`、Prometheus 暴露。  
- 下一步：增加 SLO（成功率、P95 延迟、超时率）和告警阈值。

### Q27: 如何处理长时间运行任务？
